# Architecture

## Folder structure rationale

```
src/
  app/                    # Next.js App Router — routes and API handlers only.
                          # No business logic lives here; routes call into
                          # features/ and lib/.
  features/               # Feature-based modules (fan-assistant,
                          # crowd-intelligence, operations-dashboard).
                          # Each contains its own components/, hooks/, lib/.
                          # This is the SOLID "single responsibility" boundary:
                          # a change to crowd prediction logic never touches
                          # fan-assistant code.
  lib/                    # Cross-cutting concerns shared by every feature:
    ai/                   #   provider-agnostic AI client, prompts, injection guard
    security/             #   rate limiting, sanitization, CSRF
    auth/                 #   NextAuth config
    utils/                #   generic helpers (cn, etc.)
  components/ui/          # Presentational, prop-driven, no business logic —
                          # reusable across every feature (shadcn/ui pattern).
  types/                  # Shared domain types — single source of truth so
                          # a GateReading means the same thing everywhere.
  middleware.ts           # Edge-level auth gating, runs before any route code.
tests/
  unit/                   # Pure-function tests (no DOM, no network)
  api/                    # Route handler tests with mocked AI provider
  e2e/                    # Playwright browser tests
```

Feature-based structure was chosen over type-based (`components/`, `hooks/`,
`api/` at the top level) because the modules in this brief are genuinely
independent products (a fan doesn't touch the ops dashboard code path) —
grouping by feature keeps blast radius small and lets a team split ownership
by module without merge conflicts.

## Data flow: Fan Assistant

```
Browser (ChatWindow)
  → useChat hook (client state, streaming reader)
  → POST /api/chat
      → rateLimit()               [lib/security/rateLimit.ts]
      → chatRequestSchema.parse() [lib/security/sanitize.ts]
      → guardUserInput()          [lib/ai/promptGuard.ts]
      → getAiClient().streamChat  [lib/ai/client.ts]
  ← ReadableStream<text>
  → incrementally rendered into the last assistant ChatTurn
```

## Data flow: Crowd Intelligence

```
Gate sensor / turnstile system (mocked as MOCK_GATES in HeatmapView.tsx)
  → POST /api/crowd/predict
      → crowdPredictRequestSchema.parse()
      → predictCongestion()   [deterministic model — see file for rationale]
  ← CongestionPrediction[]
  → HeatmapView renders occupancy bars + recommendation text
  → (Operations Dashboard) same predictions feed AlertsPanel aggregation
```

The congestion *number* is never generated by the LLM — see the comment
block at the top of `predictCongestion.ts` for why a safety-relevant metric
must be deterministic and auditable. The LLM's only role in this module
(via `buildCrowdInsightPrompt` in `lib/ai/prompts.ts`) is to turn the
model's output into a one-sentence narrative for a control-room operator —
narration, not computation.

## Feature → problem mapping

| Feature | Problem it solves | Why AI (vs. a static rule) |
|---|---|---|
| Fan Assistant chat | Fans can't get help in their language or find services | Natural-language, multilingual Q&A can't be hard-coded for open-ended questions |
| Crowd Intelligence prediction | Staff see congestion only after it's visually obvious | Combining occupancy + time-to-kickoff into a forward projection needs a model, even a simple deterministic one — a raw occupancy number alone doesn't tell you *when* a threshold will be crossed |
| Operations Dashboard alert narration | Control rooms get raw numbers, not prioritized action | LLM narration turns "Gate C: 91%, high" into a sentence a human can act on in one read |
| Volunteer Assistant (scaffolded) | Volunteers are first responders with no tools | Real-time translation is inherently a generative-AI task |

## Why these three modules, not all eight

The brief lists 8 possible focus areas. Building 3 to production depth (with
real security, streaming, and tests) demonstrates the pattern completely —
adding a 4th module (e.g. Sustainability recommendations) is mechanically
the same: a `features/sustainability/` folder, a route in `lib/ai/prompts.ts`,
and a page. That repetition wouldn't teach a reviewer anything new about the
architecture, so it's left as a documented extension point rather than
padding the repo with shallow copies.

## Extending: adding a new module

1. Add a folder under `src/features/<name>/{components,hooks,lib}`.
2. Add a system prompt to `lib/ai/prompts.ts` if it uses the AI client.
3. Add a route under `src/app/api/<name>/route.ts` reusing
   `rateLimit`, the relevant Zod schema, and `guardUserInput` if it accepts
   free text.
4. Add a page under `src/app/<name>/page.tsx`, code-split with
   `next/dynamic` if the module has client-only state.
5. Add unit tests for any new pure logic and an API test for the new route.
